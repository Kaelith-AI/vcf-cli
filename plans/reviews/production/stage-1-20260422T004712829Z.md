# production — Stage 1 — BLOCK

## Summary

The diff introduces significant new operational surfaces—specifically a project 'adoption' mechanism and a global registry update—without any accompanying operational definition, ownership markers, or failure contracts. While the code is functionally complete and tested, it lacks the 'operational identity' required for Production Stage 1. There is no documentation on who owns the global registry (`~/.vcf/vcf.db`), how it is backed up, or what the recovery path is if the global registry becomes corrupted during a project adoption. Furthermore, the 'adopt' feature changes the provenance of projects (marked by the `adopted` flag), but there is no defined 'failure contract' for how the system should behave when an adopted project's state is inconsistent with the VCF framework's expectations.

## Findings

- **blocker** — `src/cli.ts:787` — The `runAdopt` function introduces a new operational state (`adopted=1`) and modifies a global database (`~/.vcf/vcf.db`). There is no named owner or escalation path for users encountering corruption in the global registry or failures during the adoption process.
  - required: Define an operational owner for the global registry and document the recovery path for corrupted global state.
- **blocker** — `src/db/schema.ts:194` — The addition of the `adopted` column to the `project` table changes the project's operational provenance. However, there is no documented 'failure contract' describing how the system handles adopted projects that lack the standard VCF scaffolding (AGENTS.md, etc.) when they enter the 'reviewing' state.
  - required: Document the expected behavior and acceptable failure modes for 'adopted' projects versus 'scaffolded' projects.
- **warning** — `src/cli.ts:823` — The `runAdopt` function performs a write to the global registry and a write to the local project DB. If the global write fails after the local write, the system is left in an inconsistent state. There is no defined recovery or rollback mechanism for this partial failure.
  - required: Define the failure contract for this operation: is inconsistency acceptable, or is there a required recovery step?
