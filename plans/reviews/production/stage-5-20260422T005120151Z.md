## Summary

Deployment/release posture is only partially evidenced in this diff. The release path remains gated by an external `gh` invocation and confirm token flow, but the new state-changing behavior (`ship_release` marking projects as `shipped`) lacks a tested successful release path and rollback/recovery documentation in the scoped changes. Migration v2 is versioned and simple, but the changelog claims auto-migration with no operator action while the diff provides no release gating, backup/restore, or rollback notes for existing `.vcf/project.db` files. Health/readiness and post-deploy verification are not materially strengthened here. Repeatability is improved for config-driven review execution defaults, but release confidence is still limited because success-path release verification relies on “the 0.4 tag push is the test,” which is not a repeatable control.

## Findings

- **warning** — `test/integration/ship_release.test.ts:169` — The only new release-state regression coverage exercises the failed-release path; the comment states the successful `ship_release` transition to `shipped` is verified by a real 0.4 tag push instead of an automated test. Stage 5 does not accept one successful deployment as evidence of repeatable release safety. This leaves the new release-side state transition without repeatable proof that it behaves correctly under normal deployment conditions.
  - required: Add automated success-path coverage for `ship_release` that simulates a successful `gh` release and verifies both project DB and global registry state transition to `shipped`, or document and script an equivalent deterministic release verification path in-repo.
- **warning** — `CHANGELOG.md:53` — The changelog states `Project DB migration v2` auto-migrates existing project databases on next open with 'no operator action required,' but the scoped diff does not provide corresponding recovery guidance for a bad rollout affecting existing `.vcf/project.db` files. Stage 5 requires rollback/recovery realism for state changes; auto-applied schema changes without a documented rollback path make recovery depend on hidden operator knowledge.
  - required: Document the migration execution point and an operator-usable recovery path for v2 project DB changes, including whether rollback to prior binaries is supported after the column is added and what steps on-call should take if migration or subsequent startup fails.
- **warning** — `src/tools/ship_release.ts:243` — `ship_release` now mutates durable project state to `shipped` after a successful release and suppresses failures updating the global registry cache as non-fatal. That may be acceptable architecturally, but the operational consequence is release-state divergence between project DB and registry during or after deploy, and the diff includes no release note/runbook telling operators how to detect or repair it.
  - required: Document post-release verification and repair steps for project/global state divergence, or add deterministic reconciliation/alerting so rollback and recovery do not depend on tribal knowledge.
- **warning** — `src/db/schema.ts:203` — Migration v2 adds a durable schema field to `project` state, but the diff contains no release gating or compatibility note proving older binaries tolerate the migrated DB during rollback. If rollback after deployment restores a pre-v2 binary, Stage 5 requires explicit evidence that rollback is still executable rather than assumed.
  - required: State and test backward-compatibility expectations for pre-v2 binaries reading a v2 `project.db`, or declare rollback limitations and provide a documented recovery procedure.
