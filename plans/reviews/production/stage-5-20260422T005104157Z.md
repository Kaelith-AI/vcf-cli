# production — Stage 5 — NEEDS_WORK

## Summary

The release is generally well-structured, but there is a critical gap in the data migration path. The introduction of a schema change (v2: `adopted` column) is handled via an implicit 'auto-migrate on next open' pattern. This couples data migration to application startup/runtime, which is a high-risk pattern in production environments (e.g., multi-instance deployments or race conditions). Additionally, while the `templatesDir` fix addresses a production bundling issue, the lack of a documented rollback procedure for the database migration means a failed migration could leave the project in an inconsistent state without a clear recovery path.

## Findings

- **blocker** — `src/db/schema.ts:198` — Implicit migration: The changelog states 'Existing project.dbs auto-migrate on next open'. This couples schema evolution to the application runtime. In a production scenario with multiple instances or concurrent access, this can lead to race conditions or partial migrations. Migrations should be executed as a controlled, separate pre-deploy step.
  - required: Decouple migration execution from the application startup path. Provide a dedicated migration tool or script that can be run as a gated pre-deployment step.
- **warning** — `CHANGELOG.md:48` — Lack of documented rollback for schema change: The diff introduces a schema migration (v2) but provides no corresponding 'down' migration or documented recovery path if the `ALTER TABLE` fails or causes issues. Rollback is currently 'fiction' (reverting code does not revert the database schema).
  - required: Document the specific recovery path for the database migration. If the migration is irreversible, explicitly mark it as such and define the operational risk.
- **info** — `src/util/templates.ts:35` — The `findPackageRoot` logic is a fragile improvement over the hardcoded path. While it solves the immediate bundling issue, it relies on a loop that searches for `package.json` up to 8 levels. This is a standard pattern but should be the primary verification point for the smoke tests mentioned in the CHANGELOG.
  - required: Ensure the regression guard (followup #31) specifically tests the bundled artifact, not just the source code.
