# production — Stage 8 — BLOCK

## Summary

The diff introduces a new persistent data store (the `adopted` flag in `project.db` via a schema migration) and a new global registry update mechanism (`vcf adopt`), but provides zero evidence of backup, restore, or recovery paths for these SQLite databases. While the project is a CLI tool, the `project.db` and `vcf.db` are the authoritative sources of truth for project state and portfolio tracking. The introduction of a schema migration (v2) without a documented rollback or restore-test for the database file itself constitutes 'backup theater' by omission—assuming the file exists on disk is not a recovery plan. Specifically, the `vcf adopt` command modifies both a local and a global database without any transactional safety or recovery path if the process is interrupted mid-update, potentially leaving the global registry and local project state out of sync.

## Findings

- **blocker** — `src/db/schema.ts:198` — Introduction of Project DB migration v2 (`adopted` column). There is no documented recovery path or tested restore procedure for `project.db` if the migration fails or corrupts the database. In a production-readiness context, a schema change to a persistent store without a verified rollback/restore drill is a critical durability gap.
  - required: Provide evidence of a tested restore procedure for the SQLite databases and a documented rollback plan for the v2 migration.
- **blocker** — `src/cli.ts:798` — The `runAdopt` function performs a sequence of writes to two different databases (`project.db` and `vcf.db`) across different file system paths. If the process is killed between lines 830 and 840, the local project is marked as adopted, but the global registry is not updated (or vice versa). There is no recovery mechanism to reconcile these disparate stores of truth.
  - required: Implement a transactional or idempotent reconciliation mechanism, or document the recovery path for an operator to fix a partially adopted project.
- **warning** — `src/cli.ts:815` — The `runAdopt` function opens and closes databases manually. While SQLite is robust, the lack of a defined backup window or snapshotting strategy for the global `vcf.db` (located in the home directory) means that a corruption of this file during a high-volume `adopt` operation would result in total loss of the portfolio registry.
  - required: Define the backup/restore strategy for the global registry `vcf.db`.
