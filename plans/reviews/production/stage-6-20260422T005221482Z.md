# production — Stage 6 — NEEDS_WORK

## Summary

Reliability posture is mixed. The diff improves one hard failure mode (`templatesDir()` no longer points at the wrong location in packaged installs) and preserves shipped-state transition on failed releases. However, key resilience paths introduced or changed here are not fully bounded or observable: `ship_release` now mutates project state and then swallows global-registry update failures with no signal, creating split-brain metadata after partial failure; `runAdopt` performs multi-step local/global DB mutation without a rollback or compensating action, so interruption can leave partially adopted projects registered inconsistently. Containment is otherwise acceptable for the scoped changes because no retry loops or worker fan-out were introduced, but restart/partial-failure behavior is not fully visible enough to pass Stage 6 cleanly.

## Findings

- **warning** — `src/tools/ship_release.ts:249` — Successful release now performs a second state mutation against the global registry, but failures in that path are silently swallowed (`catch { /* non-fatal */ }`). Under partial failure, the authoritative project DB moves to `shipped` while the registry cache may remain stale with no log, metric, or operator-visible signal. This is a resilience/containment issue because restart or dependency failure creates hidden split-brain state that operators cannot detect during incident triage.
  - required: Emit an explicit warning/audit event when `setProjectState` fails, including project root and intended state, or enqueue a visible repair path. If the registry is intentionally best-effort, document and signal the degraded state rather than swallowing it silently.
- **warning** — `src/cli.ts:814` — `runAdopt` performs a multi-step adoption flow across the per-project DB and the global registry without transactional containment across those boundaries. If the process crashes or the global DB write fails after the project DB insert/update succeeds, the project is left locally adopted but not globally registered. The command logs success only after both steps, but there is no compensating rollback, retry, or explicit recovery instruction for the partial-failure case.
  - required: Add explicit failure handling for the global registry write path: either rollback the fresh local adoption when registry registration fails, or surface a clear recovery command/runbook in the error path so operators can deterministically reconcile partial adoption after interruption.
- **warning** — `src/util/templates.ts:27` — The new package-root walk improves packaged-install resilience, but the fallback path preserves the old two-level assumption when `package.json` is not found. In exotic runtime layouts this can still resolve the wrong templates directory and reintroduce the same failure mode, with no validation at resolver init time that the resolved `templates/` directory actually exists. This is bounded to template-dependent commands, but failure remains late-bound and operationally opaque.
  - required: Validate that the final resolved templates directory exists before returning it, and fail with a deterministic error naming the searched root(s). That keeps the failure contained and diagnosable instead of depending on a later `ENOENT` from template reads.
