## Summary

The changes introduce a new project adoption flow, a configuration-based model/endpoint routing system, and a fix for template resolution in bundled environments. From a reliability and resilience perspective, the changes are low-risk. The adoption flow is idempotent and handles existing state preservation. The configuration routing adds a validation layer at load-time to prevent runtime endpoint failures. The template resolution fix replaces a fragile hard-coded path with a dynamic walk-up, increasing the system's resilience to different installation layouts. No new complex async workers, retry loops, or critical data-path mutations were introduced that would create cascading failure risks.

## Findings

- **info** — `src/cli.ts:815` — The `runAdopt` function performs multiple database operations across two different databases (project.db and vcf.db). While not wrapped in a global transaction (which is impossible across separate SQLite files), the sequence is designed such that the project-local state is updated first. A failure in the global registry update is caught in a `finally` block to ensure the global DB is closed, but the project remains 'adopted' locally. This is an acceptable operational trade-off for a CLI tool where the user can simply re-run the command to synchronize the global registry.
- **info** — `src/tools/ship_release.ts:245` — The transition of project state to 'shipped' is gated on the success of the `gh` command. This correctly prevents the system from marking a project as shipped when the actual release process failed, ensuring the state reflects reality rather than intent.
