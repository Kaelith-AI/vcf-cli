# production — Stage 2 — NEEDS_WORK

## Summary

Architecture/deployment shape remains mostly single-process CLI/MCP with per-project SQLite state under .vcf/project.db plus a global registry in ~/.vcf/vcf.db. The diff adds a new adoption path that writes project state into both stores, extends project schema with an adopted flag, and makes review execution depend on config.yaml defaults when endpoint/model args are omitted. Main interaction paths are traceable, but the deployment/state model has an unresolved cross-store consistency gap: successful release now mutates project state in the project DB and only best-effort mirrors to the global registry, while adoption writes both stores separately as well. That split-state design is operationally survivable only if the cache can be rebuilt or a repair path is documented; this diff shows neither. In addition, the new project adoption surface has no accompanying operator runbook or ownership/escalation evidence in the scoped change, so restart/recovery and drift handling remain implicit. No hard in-memory durability illusion was introduced, but state-boundary clarity is incomplete enough to block confident handoff to later deploy/recovery stages.

## Findings

- **warning** — `src/tools/ship_release.ts:249` — State is now split across the authoritative per-project DB and the global registry cache, but the mirror update is explicitly best-effort and non-fatal. On a successful release, project.state is changed to 'shipped' in .vcf/project.db, then setProjectState() against the global registry is attempted and ignored on failure. This creates an architecture-level drift path between two persistent stores without an in-diff repair/rebuild mechanism or documented operational contract for which store wins during portfolio queries and operator triage.
  - required: Document and implement the recovery path for global registry drift from project DB state. Either make cache rebuild/repair an explicit supported operation with a runbook, or ensure readers can deterministically reconcile from the per-project DBs when cache writes fail.
- **warning** — `src/cli.ts:787` — The new adopt command introduces another dual-write path: it creates/updates .vcf/project.db and then separately upserts the global registry. If the second write fails after the first succeeds, the project is locally adopted but absent or stale in the registry. The architecture therefore depends on operator understanding of partial adoption outcomes, but no deployment/runbook evidence in the diff explains detection, retry, or reconciliation.
  - required: Add operator-facing recovery guidance and/or transactional reconciliation tooling for partial adoption. At minimum, document how to detect a project adopted locally but missing/stale in ~/.vcf/vcf.db and how to safely re-run or repair registration.
- **warning** — `test/integration/ship_release.test.ts:170` — The release-state transition adds a critical lifecycle mutation but only the negative path is tested here. The comment says the positive path is verified by real releases, which is not a reviewable or repeatable deployment contract. For an architecture stage, restart/fault confidence is reduced because the only automated evidence in scope is 'failed release leaves state unchanged'; there is no scoped automated proof that a successful release updates both state stores coherently.
  - required: Add an automated success-path test for ship_release state transition and global registry mirroring using a controlled gh success stub, so later production stages can rely on repeatable evidence instead of ad hoc real-release validation.
- **warning** — `src/cli.ts:1738` — A new production-facing project adoption surface is added, but the scoped diff contains no runbook, ownership, or support documentation covering its state model ('adopted=1', default reviewing state, re-adoption preservation, partial scaffolding omission). Stage 2 requires deployment/state behavior to be understandable by the inheriting operator; in this diff the human path is encoded in CLI description and changelog only.
  - required: Add or link an operator runbook/doc for adoption covering what files/databases are created, what 'adopted' means operationally, re-adoption behavior, expected state defaults, and how to recover from interrupted/partial adoption.
