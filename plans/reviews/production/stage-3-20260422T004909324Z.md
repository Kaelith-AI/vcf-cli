# production — Stage 3 — NEEDS_WORK

## Summary

The diff introduces critical operational changes—specifically a new project adoption flow (`vcf adopt`) and a dynamic configuration resolution system for LLM endpoints—but lacks the telemetry required to diagnose failures in these paths. While the fix for `templatesDir()` is a significant stability improvement, the new features are 'silent' from an operational perspective. There is no instrumentation to track the success/failure rate of project adoptions, no logging of the resolved endpoint/model for the new defaults system, and no visibility into the DB migration process for existing users.

## Findings

- **warning** — `src/cli.ts:787` — The `runAdopt` function performs several critical side effects (creating directories, writing to two different databases) but only uses basic `log()` calls. If a user reports a failure during adoption, there is no structured telemetry to distinguish between a filesystem error, a project DB write failure, or a global registry update failure.
  - required: Add structured logging or telemetry events for each major step of the adoption process, including the specific failure point.
- **warning** — `src/tools/review_execute.ts:104` — The new endpoint resolution logic (explicit arg → defaults → legacy) is a high-risk area for 'silent' misconfiguration. The code resolves the `endpointName` and `modelId`, but it does not log which resolution path was taken. An operator debugging a 'wrong model' issue would have to guess whether the user provided an arg, a default was hit, or the legacy fallback occurred.
  - required: Log the resolved endpoint and model, explicitly stating the source of the resolution (e.g., 'Resolved endpoint via config.defaults.review').
- **info** — `src/db/schema.ts:199` — The DB migration v2 adds a column to the `project` table. There is no logging or telemetry indicating when a migration is successfully applied to a local project DB, making it difficult to verify if a user is running the latest schema version during triage.
  - required: Add a log entry when a migration is successfully applied.
- **info** — `src/cli.ts:824` — The `adopt` command's error handling simply catches and prints the error message. For operational debugging, this lacks the context of the input parameters (path, name, state) that led to the failure.
  - required: Include the input arguments in the error log when `runAdopt` fails.
