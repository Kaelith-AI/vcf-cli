# production — Stage 7 — BLOCK

## Summary

Operational readiness is not credible for the new adoption path or the changed release-state behavior. The diff introduces a new operator-facing command (`vcf adopt`) and changes release state transitions, but provides no runbook, no ownership/escalation artifact, no recovery guidance for partial adoption/registry divergence, and no operator procedure for diagnosing or correcting state mismatches. A non-author on-call inheriting this tonight would need to read source to understand what files are created, what state is preserved, how to recover from a failed/adverse adoption, or how to reconcile local/global DB drift after `ship_release`. Stage 7 blocks on missing durable human-operability artifacts for these new operational surfaces.

## Findings

- **blocker** — `src/cli.ts:1655` — New operator-facing `adopt` command is introduced with only inline CLI description; the scoped diff contains no runbook or operations doc covering when to use adoption, prerequisite checks, expected filesystem/database side effects, idempotent re-run behavior, or how to recover from a partially completed adoption. This fails the stage requirement that a non-author operator be able to run and troubleshoot the service using durable operational artifacts rather than source spelunking.
  - required: Add a runbook entry for project adoption covering: prechecks, exact command usage, created artifacts (`.vcf/project.db`, global registry row), state defaults/preservation semantics, verification steps, and rollback/remediation steps for failed or mistaken adoption.
- **blocker** — `src/cli.ts:787` — `runAdopt` mutates two state stores in sequence (project-local DB, then global registry) without an operator procedure for detecting and repairing divergence if the second step fails or if the command is interrupted between them. The code logs success paths, but there is no documented human recovery path for 'local adopted, global registry missing/stale' or wrong name/state selection. For a new support surface touching project inventory/state, lack of a documented recovery path is an operational blocker.
  - required: Document a reconciliation procedure for adoption failures and partial success: how to inspect local `project` row, how to inspect/update the global registry entry, when re-running `vcf adopt` is safe, and how to undo or correct an accidental adoption.
- **blocker** — `src/tools/ship_release.ts:240` — Release behavior now changes `project.state` to `shipped` and attempts to mirror that into the global registry, but registry update failures are silently swallowed as 'non-fatal'. The diff adds no operator-facing guidance for noticing or correcting local/global state drift after release. This means a non-author responder cannot reliably reconcile portfolio status discrepancies without reading source or ad hoc DB inspection.
  - required: Add operational documentation for post-release verification and drift repair: how to verify local project state vs global registry cache, expected behavior when registry mirroring fails, and the supported manual repair command/procedure.
- **warning** — `src/db/schema.ts:197` — Migration v2 adds `project.adopted`, but the changelog's 'no operator action required' note is not a usable support artifact. There is no runbook guidance for how operators verify migration success, identify migration-related startup issues, or recover if an existing `project.db` fails to migrate on open. For a DB schema change affecting all existing projects, this leaves non-authors without a first-response procedure.
  - required: Document migration verification and recovery steps in the operations/runbook material, including how to confirm schema version/adopted column presence and what to do if project DB open/migration fails.
- **warning** — `src/tools/review_execute.ts:97` — Review execution now depends on optional `config.defaults.review.endpoint/model` fallback, but the only operator guidance in scope is commented seed text in `vcf init`. There is no durable troubleshooting doc for the new validation path (`endpoint not provided and config.defaults.review.endpoint is unset`) or for precedence rules during incident/debugging. This increases source-archaeology burden for support staff diagnosing review failures.
  - required: Add support documentation for endpoint/model resolution order, common failure messages, and the operator checklist for validating `config.yaml` when review execution fails due to missing or mismatched defaults.
- **warning** — `CHANGELOG.md:9` — Operationally significant behavior is described only in the changelog. Changelog notes are not a substitute for runbooks, escalation paths, or incident procedures, and they are not sufficient evidence that humans other than the builder can operate the new surfaces under pressure.
  - required: Promote operator-relevant procedures from release notes into durable support documentation referenced by the service's operational docs.
