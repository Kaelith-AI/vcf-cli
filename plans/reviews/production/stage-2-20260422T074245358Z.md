# production — Stage 2 — NEEDS_WORK

## Summary

Artifact class remains CLI/tooling, not a pager-backed service. The deployed shape is operationally coherent as a single-user local tool: state lives in local SQLite files (`<project>/.vcf/project.db`, `~/.vcf/vcf.db`, `~/.vcf/lessons.db`), the MCP server is single-process, and outbound LLM traffic is explicit through configured endpoints. The main Stage-2 issues are both state-boundary truth problems around the new lessons mirror: (1) docs claim operators can disable the cross-project mirror with `config.lessons.global_db_path: null`, but the shipped schema/tool path does not support `null`; and (2) the project DB write succeeds even when the global mirror write fails, leaving the cross-project store as a best-effort cache with no replay/reconcile path. For a CLI this is not a BLOCK, but it is a meaningful architecture/state-boundary gap that later stages should not inherit as if the mirror were a reliable, operator-controllable boundary.

## Findings

- **warning** — `src/config/schema.ts:257` — `config.lessons.global_db_path` is modeled as optional string-only config, but the release docs describe `null` as the operator escape hatch to disable the global lessons mirror. That means the documented state-boundary control for cross-project lesson storage does not actually exist in the deployed shape. This is an architecture/deployment contradiction: operators are told they can turn off the global mirror, while `lesson_log_add` still resolves and opens a DB path whenever lessons are written.
  - required: Either implement `null` as a first-class 'mirror disabled' state end-to-end (schema, resolver, `lesson_log_add`, `lesson_search`, docs), or remove the `null` claim from README/CHANGELOG and carry the residual explicitly: global lesson mirroring is always on when lesson logging is used.
- **warning** — `src/tools/lesson_log_add.ts:104` — The lessons state boundary is split across two SQLite stores, but writes are not survivable as a coherent dual-write: the project DB insert happens first, then the global mirror write is attempted and any failure is downgraded to `global_mirror_error` while the tool still returns success. Operationally this makes `~/.vcf/lessons.db` a best-effort cache, not a reliable mirror, and there is no replay/reconcile artifact in the diff to heal missed global writes after permissions/disk failures.
  - required: Make the mirror contract honest and recoverable. Either: (a) document the global lessons DB as non-authoritative best-effort state and add a reconcile/resync command or queued retry path; or (b) make the dual-write atomic enough for the intended contract. Later stages should treat `lesson_search scope=global|all` as lossy until a repair path exists.
