# production — Stage 4 — NEEDS_WORK

## Summary

Artifact class remains developer CLI plus local MCP server, so Stage-4 service-on-call capacity bars do not apply. The main new runtime surfaces are local SQLite-backed lesson logging/search and lifecycle-report generation. `lifecycle_report` is mostly bounded: section count is fixed, audit/recent row caps are configurable (`src/config/schema.ts`), and narrative mode uses one LLM call per non-project section with an overall timeout. The material capacity gap is `lesson_search`: the implementation reads entire project/global lessons tables into memory and filters/ranks in-process, which does not match the explicit 10k-entry / p95<100ms scale claim in the plan and tests. No other blocking unbounded resource path was introduced in this diff.

## Findings

- **warning** — `src/tools/lesson_search.ts:72` — `lesson_search` does a full-table read from both project and global lessons stores, then filters, de-dups, parses tags JSON, scores, sorts, and limits in memory. The hot path is `readLessons()` with `SELECT ... FROM lessons` and no WHERE/LIMIT pushdown (`src/tools/lesson_search.ts:155` and `src/tools/lesson_search.ts:169`). This is an evidence-vs-claim gap against the stated 10k-entry target in `plans/phase-2-inward-loop-plan.md`: one large global lessons DB or `scope=all` query can dominate latency and memory even when the caller asks for a small `limit`.
  - required: Push filtering and bounds into SQLite: add SQL predicates for `stage` and coarse text search where possible, cap candidate rows before materializing, and avoid unconditional full-table reads for `scope=global`/`all`. If full-text search is intentionally deferred, carry that forward explicitly as accepted residual capacity risk and narrow the published scale claim until measured evidence exists.
