# code — Stage 6 — NEEDS_WORK

## Summary

Platform compatibility is overstated by the new CLI adopt flow and partially under-verified in tests. The scoped diff adds a user-facing command that claims to adopt an existing project directory via an absolute path, but the implementation derives the default project name using a hard-coded '/' split, which breaks on native Windows paths. The command description and changelog present this as a general workflow for existing projects, while the code currently only behaves correctly on POSIX-style paths or Windows paths passed with forward slashes. Separately, ship_release’s success path now mutates project state based on gh success, but the added test explicitly defers positive-path verification to a real release instead of an automated test, leaving environment/runtime behavior unproven. The rest of the config-defaults and template-root changes look directionally compatible, with config validation catching undeclared endpoints and template resolution becoming less layout-sensitive.

## Findings

- **warning** — `src/cli.ts:816` — The new adopt command computes the default project name with `absRoot.split("/").pop()`, which assumes POSIX separators. On native Windows paths resolved like `C:\work\repo`, this returns the whole path instead of the basename, so default naming/slugging becomes incorrect despite the command being presented as a general existing-project adoption workflow.
  - required: Use platform-safe path handling for basename extraction, e.g. Node `path.basename(absRoot)`, so default naming works on Windows as well as POSIX.
- **warning** — `src/cli.ts:1657` — The CLI help text for `adopt` states it adopts an existing project directory without documenting any OS/path limitations, but the implementation has a path-separator assumption that makes native Windows usage fragile. Stage 6 requires support claims/docs to match reality; either the command must be made platform-safe or the support boundary must be narrowed explicitly.
  - required: After fixing basename handling, keep the current broad help text. If not fixing it, document that native Windows paths are not supported and require a POSIX-style environment such as WSL.
- **warning** — `test/integration/ship_release.test.ts:171` — The added test coverage explicitly leaves the successful `ship_release` state transition untested in automation and relies on 'the 0.4 tag push is the test'. That is weak environment evidence for a feature whose behavior depends on external tooling/runtime context (`gh` availability/auth/success). For platform and environment compatibility, later stages should not inherit confidence from a release-path mutation that is only manually/prod verified.
  - required: Add an automated positive-path test by stubbing/spying the release command success path so the `project.state -> shipped` transition is verified in CI without requiring a real authenticated GitHub release environment.
