# code — Stage 4 — NEEDS_WORK

## Summary

The implementation of the `adopt` command and the `templatesDir` resolution logic contains critical correctness and failure-handling flaws. Specifically, the `adopt` command fails to close the project database connection in the success path, leading to potential file locks on Windows or resource leaks. Additionally, the `templatesDir` fix introduces a constant `PACKAGE_ROOT` that is redacted in the diff, but the logic provided in `findPackageRoot` is not actually used to initialize that constant, creating a disconnect between the logic and the implementation. The configuration defaults and `ship_release` state transitions appear correct.

## Findings

- **blocker** — `src/cli.ts:818` — The `pdb` (Project Database) connection is opened via `openProjectDb` but is only closed if the logic reaches the end of the function. However, the `upsertProject` call to the global database is wrapped in a try/finally block, but the `pdb` connection itself is not. If `upsertProject` or any logic before the {finally} block throws, the `pdb` connection remains open.
  - required: Wrap the `pdb` operations in a try/finally block to ensure `pdb.close()` is called regardless of success or failure.
- **warning** — `src/cli.ts:804` — The `runAdopt` function uses `absRoot.split('/').pop()` to determine the project name. This is fragile and will fail on Windows paths (which use backslashes) or if the path ends with a trailing slash, resulting in an empty string as the name.
  - required: Use `path.basename(absRoot)` instead of manual string splitting.
- **blocker** — `src/util/templates.ts:38` — The `PACKAGE_ROOT` constant is assigned a redacted value in the diff. However, the `findPackageRoot` helper function is defined immediately above it, but is not called to initialize `PACKAGE_ROOT`. This suggests either the constant is hardcoded (which defeats the purpose of the fix) or the logic is missing from the final code. If `PACKAGE_ROOT` is a static string, the `templatesDir()` function will fail in different environments.
  - required: Initialize `PACKAGE_ROOT` by calling `findPackageRoot(__dirname)`.
- **info** — `src/cli.ts:814` — The `runAdopt` function performs an `INSERT` or `UPDATE` on the project database, but does not wrap these in a transaction. While it is a single statement, the subsequent update to the global registry is a separate operation. A failure in the global registry update could leave the project database in an 'adopted' state while the global registry is out of sync.
  - required: Consider if this atomicity is required; if not, the mark as 'info' for consistency.
