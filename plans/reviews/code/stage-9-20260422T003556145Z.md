## Findings

- **warning** — `src/cli.ts:815` — `runAdopt` derives the default project name with `absRoot.split("/").pop()`, which is not path-platform safe. On Windows-style paths, the split will not separate path segments, so adoption can record the full absolute path as the project name and then slugify that malformed value into the global registry. This is a release-readiness issue because `vcf adopt` is explicitly positioned for real operator use against existing projects, and mis-registration is operator-visible state that is harder to remediate after release.
  - required: Use a path-aware basename helper from `node:path` instead of manual `"/"` splitting, and add an automated test covering Windows-style input or basename extraction behavior.
- **warning** — `test/integration/ship_release.test.ts:145` — The new `ship_release` behavior mutates durable project state to `shipped` and mirrors that into the global registry on success, but the diff only adds a negative-path test. The comment says the positive path is verified by a real release/tag push, which is not a bounded automated regression guard. For a release-stage review, rollback/remediation confidence is reduced when a persisted state transition is only partially tested and the success path depends on manual or environmental validation.
  - required: Add an automated success-path integration test that stubs the release command/result and verifies both `project.state='shipped'` and the global registry cache update, so future regressions in release-state handling are caught before shipping.
