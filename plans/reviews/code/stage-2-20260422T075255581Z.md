# code — Stage 2 — NEEDS_WORK

## Summary

System shape is mostly coherent: new shared adoption core (`src/project/adopt.ts`) pulls CLI and MCP adoption onto one path; lesson logging/search, lifecycle reporting, and reviewer-overlay resolution are wired as real subsystems rather than folder theater; and the review flow now threads carry-forward/response-log context on live paths. Two boundary issues keep this from a clean PASS: `review_execute` now bypasses the prepared run snapshot for reviewer config/overlay loading, which weakens the prepare→execute contract, and `project_init_existing` still registers its MCP schema via `.shape`, leaving this new tool outside the stricter tool-boundary pattern used elsewhere in the same phase.

## Findings

- **warning** — `src/tools/review_execute.ts:278` — `review_execute` no longer executes solely from the artifacts prepared into `runDir`; `composeMessages()` prefers the live KB-loaded reviewer base/overlay (`overlay?.base`) over the snapshot in the prepared workspace. That makes a prepared run non-self-contained: changing KB reviewer files between `review_prepare` and `review_execute` can change the execution prompt for the same `run_id`, which weakens the system boundary between preparation and execution and makes runs harder to reason about/reproduce.
  - required: Snapshot the resolved reviewer base+overlay during `review_prepare` and have `review_execute` read only the prepared copies for that `run_id`, or persist an explicit versioned reference proving why live KB reads are safe for an existing prepared run.
- **warning** — `src/tools/project_init_existing.ts:79` — The new adoption MCP tool registers `inputSchema: ProjectInitExistingInput.shape` instead of the full strict schema object. This leaves the tool-surface boundary inconsistent with the phase's other formalized tools (`lesson_log_add`, `lesson_search`, `response_log_add`), because unknown keys can bypass the SDK-layer strict-object rejection that the rest of this bundle is standardizing on.
  - required: Register the full `ProjectInitExistingInput` object as `inputSchema`, not `.shape`, so the adoption tool participates in the same strict MCP boundary as the other new tools.
