# code — Stage 3 — NEEDS_WORK

## Summary

The changes introduce critical functionality (project adoption, config defaults, and a fix for template resolution) and include corresponding tests. However, the fix for `templatesDir()` in `src/util/templates.ts` is fundamentally broken in the provided diff due to a redaction/placeholder error, and the 'adopt' functionality lacks a regression test for the idempotency claim made in the CHANGELOG.

## Findings

- **blocker** — `src/util/templates.ts:35` — The implementation of `templatesDir()` uses a constant `PACKAGE_ROOT` which is defined as `[REDACTED]`. This is a non-functional placeholder. The logic intended to find the package root via `findPackageRoot` is defined but never actually called. The function `templatesDir()` simply returns a path relative to a redacted constant, which will cause an immediate `ENOENT` or crash in any environment.
  - required: Call `findPackageRoot(__dirname)` to dynamically resolve the package root instead of using a redacted constant.
- **warning** — `src/cli.ts:787` — The `runAdopt` function implements a project adoption flow. The CHANGELOG claims this is 'idempotent re-adoption preserves existing state + name'. However, there are no integration tests verifying this idempotency. Given that this modifies both a local `.vcf/project.db` and a global registry, the risk of state corruption or accidental overwrite of existing project metadata is high.
  - required: Add an integration test in `test/integration/` that adopts a project, verifies the state, and then adopts it again with different parameters to ensure existing state is preserved as claimed.
- **info** — `src/cli.ts:815` — The `runAdopt` function uses `absRoot.split('/').pop()` as a fallback for the project name. This is platform-dependent and will fail on Windows (where the path separator is `\`).
  - required: Use `path.basename(absRoot)` instead of manual string splitting.
