# code — Stage 4 — NEEDS_WORK

## Summary

Core paths reviewed: CLI adoption flow, review_execute default resolution, ship_release state transition, template path resolution, and config/schema migration wiring. The review/defaults and ship_release changes look materially correct on their main paths, but the new adopt flow has correctness gaps that will mis-handle valid filesystem paths and can leave the per-project DB and global registry out of sync on registry write failure. Confidence is medium because the diff adds behavior without corresponding adopt-path tests.

## Findings

- **warning** — `src/cli.ts:814` — Default project name derivation is not path-semantically correct on Windows or for paths with trailing separators. `absRoot.split("/").pop()` only works for POSIX-style separators and can also produce an empty string for inputs ending in `/`, causing adoption to fall back to the generic name `project` instead of the actual directory name. That yields incorrect slugs/registry names for a valid local path.
  - required: Use path-aware basename extraction (for example Node's `basename(absRoot)`) after normalizing the resolved path, so both Windows paths and trailing-separator inputs produce the expected directory name.
- **warning** — `src/cli.ts:818` — `runAdopt` writes the project DB first and only then updates the global registry. If `upsertProject` fails, the command exits with an error after having already created/updated `.vcf/project.db` with `adopted=1`, leaving local state changed but the global registry missing or stale. The changelog and command behavior imply adoption should register the project as part of the operation; this partial-commit failure mode is not handled credibly.
  - required: Make the operation failure-safe: either validate registry writability before mutating the project DB, or add rollback/compensation when global registration fails, or explicitly persist and surface a recoverable degraded state instead of returning a generic failure after partial success.
- **warning** — `test/integration/ship_release.test.ts:171` — The new ship_release behavior updates both the project DB and global registry on successful release, but the added test only covers the failure path and explicitly defers positive-path verification to a real release. That leaves the critical success-path state transition and registry mirroring unverified in automated tests, reducing confidence in correctness for a stateful flow.
  - required: Add an automated positive-path test that stubs the release command to succeed and asserts both `project.state='shipped'` in the project DB and the mirrored global registry state cache update.
