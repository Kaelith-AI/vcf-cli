# code — Stage 8 — NEEDS_WORK

## Summary

Sensitive data in scope here is primarily project filesystem metadata (absolute project paths, names, state) plus review/model-routing configuration that can direct project content to external endpoints. Main trust-relevant flows added in this diff: `vcf adopt` persists an arbitrary local directory path into both per-project and global SQLite DBs and logs it to stdout; `review_execute` now permits endpoint/model selection via config defaults, which can silently route review content to an external provider if configured. Overall handling confidence is medium-low: no obvious raw secret logging was added, but there is a meaningful trust-signal gap around external model-provider disclosure and one portability issue that can mis-handle adopted project names/paths. Security Stage 1 should review model-provider disclosure/transfer expectations and retention implications of storing absolute project roots in the global registry.

## Findings

- **warning** — `src/tools/review_execute.ts:100` — `review_execute` now silently falls back to `config.defaults.review.endpoint` / `model` when the caller omits explicit routing. That changes the disclosure boundary for review content: a run can send project review material to whichever endpoint the operator configured without any per-call indication or confirmation. For a trust/compliance stage, model-provider transfer is a first-class boundary and this implementation adds hidden disclosure risk, especially when the configured endpoint is remote or public.
  - required: Expose the resolved endpoint/model clearly in a user-visible preflight/result path before transmission, and document/guard that omitted `endpoint` means review content may be sent to the configured provider. At minimum, add an explicit audit/log/trust signal that default routing is being used; preferably require confirmation when the resolved endpoint trust level is public or non-local.
- **warning** — `src/cli.ts:806` — `runAdopt` derives the default project name with `absRoot.split("/").pop()`. This is not path-platform-safe and can preserve the full absolute path as the name on Windows-style inputs. That increases unnecessary exposure because the project name is then persisted and logged, potentially duplicating sensitive path information that should have been minimized to a basename.
  - required: Use `basename(absRoot)` from `node:path` instead of string-splitting on `/`, so only the directory name is stored/logged across platforms.
