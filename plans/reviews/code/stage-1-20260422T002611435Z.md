Repo appears reviewable enough for deeper stages, but there is one material truth/reviewability issue in the new adopt path: deriving the default project name from `absRoot.split("/")` is not portable and will misidentify Windows paths, which undermines the claimed ability to adopt existing projects generically.

## Findings

- **warning** — `src/cli.ts:815` — `vcf adopt` claims to adopt an existing project directory by defaulting the project name to the directory basename, but the implementation uses `absRoot.split("/").pop()`. This only works for POSIX-style paths. On Windows, a normal path like `C:\work\myproj` will not split, so the default name becomes the entire path string, feeding a malformed slug and incorrect registry/project identity. At Stage 1 this is a truth gap in claimed cross-environment behavior for a primary entrypoint.
  - required: Use a path-aware basename extraction (`path.basename(absRoot)` or equivalent) instead of string-splitting on `/`, and add a test covering Windows-style absolute paths so the default naming behavior is evidence-backed.
