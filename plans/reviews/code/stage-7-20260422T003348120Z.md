# code — Stage 7 — NEEDS_WORK

## Summary

Overall readability is decent and the new comments generally explain intent and rollout context well. Core flows in config validation, review endpoint/model fallback, and template resolution are understandable. However, change safety is weakened by one cross-platform naming/path assumption in `runAdopt` and one bus-factor/testing gap in `ship_release`: the implementation intentionally leaves successful state transition coverage to a manual real-release path, which makes future modification of release-state behavior harder to validate safely. Maintainability is therefore acceptable but not yet strong enough for a clean pass.

## Findings

- **warning** — `src/cli.ts:817` — `runAdopt` derives the default project name with `absRoot.split("/").pop()`, which encodes POSIX path semantics inside a CLI that otherwise uses Node path helpers. On Windows this can produce the full path instead of the basename, leading to surprising names/slugs and making future path-handling changes error-prone because maintainers must notice this one-off manual split.
  - required: Use a path helper (`basename(absRoot)`) instead of manual string splitting so the naming logic matches the rest of the filesystem handling and remains portable.
- **warning** — `test/integration/ship_release.test.ts:169` — The new release-state behavior lacks an automated success-path test. The comment says the positive transition is verified only by a real `0.4` release, which creates tacit knowledge and weakens change safety around a stateful lifecycle transition. A maintainer changing `ship_release` later cannot rely on local CI to catch regressions in the shipped-state update or global registry mirroring.
  - required: Add an automated test for the successful path by stubbing the release command/process result and asserting both `project.state='shipped'` and global registry cache update behavior. Do not rely on a real release as the only verification for this branch.
