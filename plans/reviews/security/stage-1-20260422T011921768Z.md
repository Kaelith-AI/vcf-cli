## Summary

Stage 1 scope is only partially established by this diff. The new adoption surface and review routing changes expose identifiable assets and boundaries: project metadata in <project>/.vcf/project.db and ~/.vcf/vcf.db, operator-supplied filesystem paths, config.yaml endpoint/model defaults, outbound review payloads to model providers, and gh-driven release state transitions. The diff also makes some trust boundaries explicit (allowed_roots parity for CLI adopt, endpoint declaration validation, model-provider boundary in review_execute), but key Stage 1 obligation/scope signals remain under-modeled: the new external model-routing defaults can steer review content to any configured provider without any classification or disclosure of what data review payloads contain, and the adopt path can intentionally operate without config/allowed_roots when config load fails, leaving the effective trust boundary dependent on operator environment rather than an explicit invariant. Later security stages can proceed, but only after these scope assumptions are recorded more clearly.

## Findings

- **warning** — `src/cli.ts:802` — `runAdopt` intentionally skips the `workspace.allowed_roots` boundary whenever config load fails for reasons other than `E_SCOPE_DENIED` (including missing config and parse errors). For Stage 1, that leaves the filesystem trust boundary only partially defined: the MCP surface is config-gated, while the CLI surface is operator-gated unless config happens to load cleanly. The code comment documents intent, but the security invariant is still ambiguous because a malformed config silently widens scope to any operator-accessible directory.
  - required: Document this asymmetry as an explicit trust assumption/invariant for the adopt surface, or narrow the exception so only 'config missing' bypasses the boundary while parse/validation failures fail closed.
- **warning** — `src/tools/review_execute.ts:100` — `review_execute` now resolves endpoint/model from `config.defaults.review`, which means review payloads can be routed to a configured external provider without the tool call naming that provider explicitly. Stage 1 requires the model-provider trust boundary and data-handling scope to be explicit. This diff shows outbound review content is sent to whichever endpoint is configured, but there is no adjacent classification of what review payloads may contain (repo diffs, review prompts, possibly redacted sensitive content) or any obligation signal tied to provider selection/retention.
  - required: Add explicit security/privacy documentation or config-level notes identifying model providers as external trust boundaries and classifying the review data sent through `review_execute`, including retention/jurisdiction follow-up for later stages.
