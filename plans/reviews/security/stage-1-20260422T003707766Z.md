# security — Stage 1 — NEEDS_WORK

## Summary

Security scope is partly inferable from the diff: this CLI/MCP server manages project metadata in per-project and global SQLite DBs, executes review calls against configured model endpoints, and can ship releases via external tooling. New changes add adoption of arbitrary existing directories into VCF tracking, per-tool endpoint/model defaults, and a shipped-state transition. Major assets and boundaries are visible, but Stage 1 should not pass cleanly yet because the new adoption surface expands filesystem and registry trust without any documented boundary/authorization model, and external model/provider handling remains only partially scoped in the changed surfaces. Downstream security review can proceed, but with explicit follow-up on operator privilege, path trust, global registry integrity, and provider data-routing obligations.

## Findings

- **warning** — `src/cli.ts:787` — New `vcf adopt` introduces a high-impact scope expansion: any local directory can be registered into VCF, a `.vcf/project.db` can be created inside it, and the path is written into the global registry. The diff shows input validation for existence/type, but no explicit security documentation or trust-boundary statement for who is allowed to adopt which paths, whether symlinked/borrowed/shared workspaces are in scope, or what guarantees later tools may assume about adopted project contents. Stage 1 requires explicit actor/privilege and trust-boundary mapping before later review; this surface currently relies on implicit operator trust.
  - required: Document and carry forward the adoption trust model: actor class allowed to invoke `adopt`, assumptions about local filesystem trust, whether symlinks/network mounts/shared directories are supported, and what later tools may safely assume about adopted projects versus scaffolded projects. If adopted projects are intentionally semi-trusted, record that invariant explicitly for later stages.
- **warning** — `src/server.ts:158` — `project_init_existing` is now registered in the MCP server, creating a new project-bootstrap boundary for remote/tool-driven callers, but the scoped diff does not include the tool implementation or any authorization/gating details for this surface. Stage 1 cannot fully map actors and trust zones for the new MCP entry point from the provided evidence. If MCP callers can adopt arbitrary paths, this is a trust-boundary change with filesystem and registry impact.
  - required: Include the `src/tools/project_init_existing.*` implementation in scope or add an explicit boundary note describing how MCP access to project adoption is authenticated/authorized and constrained to intended operator contexts. Without that, later security stages inherit an underspecified filesystem-control surface.
- **warning** — `src/tools/review_execute.ts:97` — `review_execute` now silently routes to `config.defaults.review.endpoint` and `config.defaults.review.model` when explicit arguments are omitted. This makes provider selection less visible at call sites while still sending review content to an external or semi-trusted endpoint. The trust boundary to model providers remains present, but the new default-routing path increases the chance that sensitive review material is sent off-box based only on config. Stage 1 requires this provider/data-handling boundary to be explicit.
  - required: Record and document that review content may be transmitted to the configured default endpoint when callers omit `endpoint`, including the actor who controls `config.yaml`, the trust level of configured endpoints, and what classes of review data may cross that boundary. Carry this forward for later provider-data-handling and compliance review.
