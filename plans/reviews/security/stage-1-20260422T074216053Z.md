# security — Stage 1 — NEEDS_WORK

## Summary

Stage 1 scope is mostly explicit in this diff. High-value assets now clearly include project-local and global lesson logs, response-log history, audit/build/review metadata, lifecycle reports, and endpoint-routing config. Main entry points are MCP tools and matching CLIs (`lesson_log_add/search`, `response_log_add`, `lifecycle_report`, `project_init_existing`/`vcf adopt`). Major trust boundaries are also visible: project session ↔ global lessons DB, project data ↔ narrative/report/review LLM endpoints, CLI/operator ↔ filesystem/allowed_roots, and reviewer prompt assembly ↔ per-model overlays. Obligation signals are documented rather than enforced: residency/retention must match project classification; redaction is explicitly not confidentiality. One material scope/trust-boundary problem remains: the docs tell operators they can disable the global lessons mirror with `config.lessons.global_db_path: null`, but the shipped schema/code do not support `null` and still route lesson writes to the global DB path. That leaves a documented isolation control unavailable for confidential projects on shared workstations.

## Findings

- **warning** — `README.md:190` — The documented trust-boundary escape hatch for cross-project lesson isolation is not implemented. README tells operators to set `config.lessons.global_db_path: null` to disable the global mirror for sensitive/NDA projects, but `src/config/schema.ts:259-264` defines `global_db_path` as optional string only (not nullable), `src/db/globalLessons.ts:55-66` resolves missing/empty values to the default `~/.vcf/lessons.db`, and `src/tools/lesson_log_add.ts:108-131` always attempts the global mirror write. This creates false confidence around a named confidentiality boundary: operators may believe cross-project storage is disabled when lesson data is still mirrored.
  - required: Either implement `null` as an explicit disable state end-to-end (schema, path resolver, lesson write/read paths, docs) or remove the `null disables mirror` claim from README/CHANGELOG and document only controls that actually exist. For Stage 1 scope clarity, the effective boundary must match the shipped behavior.
