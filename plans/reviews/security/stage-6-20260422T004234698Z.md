# security — Stage 6 — NEEDS_WORK

## Summary

Stage 6 review found no direct hardcoded credential exposure in the scoped diff, and config validation was improved by schema-checking defaults.<tool>.endpoint against declared endpoints. However, the new config-driven endpoint routing in review_execute permits silent use of whatever endpoint is preconfigured in config.defaults.review, including potentially high-trust provider-backed endpoints, without any additional confirmation or environment segregation controls visible in this diff. Rotation/revocation posture for provider credentials remains implicit rather than enforced. Confidence: Medium, limited to the scoped diff.

## Findings

- **warning** — `src/tools/review_execute.ts:100` — review_execute now resolves endpoint from config.defaults.review.endpoint when the caller omits endpoint. This removes an explicit per-call trust choice and can silently route review content to whichever endpoint is preconfigured. The existing public-endpoint gate still blocks trust_level=public unless allow_public_endpoint is set, but there is no comparable control here for accidental use of private/provider-backed endpoints across environments or projects. For a review system that may handle sensitive diffs and carry-forward findings, silent config-based routing increases credential blast-radius and raises the risk of sending review payloads through an unintended model-provider identity.
  - required: Add an explicit control for default-routed review execution: e.g. surface the resolved endpoint prominently and require opt-in for non-local/provider-backed endpoints, or enforce environment/project-level policy that review defaults may only target approved endpoints. Document and test the trust boundary for config.defaults.review.endpoint so sensitive review payloads cannot be silently redirected by config alone.
- **warning** — `src/cli.ts:193` — vcf init now seeds config.yaml comments encouraging operators to place per-step endpoint/model defaults directly in project config. While the example does not contain secrets, it establishes a workflow where provider selection is project-local and persistent. In practice this often co-locates credential-bearing operational config with source-controlled files, but no companion guard is shown here for keeping real endpoint credentials out of config templates or for distinguishing local vs shared deployment paths.
  - required: Augment generated config guidance to state explicitly that config.yaml must reference named endpoints only and must never contain raw API keys or bearer tokens; pair this with setup docs or validation/tests ensuring credentials remain externalized to environment/secret stores. If such controls exist outside the diff, carry them forward explicitly in docs/tests.
