# security — Stage 2 — NEEDS_WORK

## Summary

Threat model review of the scoped diff found one credible medium-severity abuse path introduced by the new project adoption flow: an operator can adopt an arbitrary existing directory and the tool will create/modify a hidden `.vcf` subtree inside that path with no trust-boundary guard against sensitive/system-owned directories. The rest of the diff is mostly bounded: review endpoint/model defaulting preserves the existing public-endpoint gate in `review_execute`, and the shipped-state transition is gated on successful release. AI-specific threat categories were evaluated against the changed surfaces; most remain structurally unchanged by this diff, but prompt/provider/tool trust still depends on downstream review execution controls rather than hard isolation in the adoption path.

## Findings

- **warning** — `src/cli.ts:793` — `vcf adopt` will operate on any existing directory path and unconditionally create `absRoot/.vcf` plus a `project.db` inside it after only checking existence and `isDirectory()`. There is no denylist/guard for high-risk targets such as `/`, a home directory, VCS-controlled third-party checkouts, or other sensitive existing trees. Abuse path: attacker convinces a privileged operator to run `vcf adopt <sensitive-dir>`; tool writes hidden state into that directory and registers it globally; later VCF review or lifecycle tools may treat that path as an in-scope project and read/process unrelated files under an incorrect trust model. This is not an immediate RCE/auth bypass, but it is a realistic tampering/trust-boundary issue introduced by the new adoption feature.
  - required: Add structural path-safety checks before adoption: reject dangerous roots (e.g. filesystem root, home dir, parent of many projects, existing VCF workspace internals), require explicit force/confirm for non-empty directories or directories lacking project markers, and document the trust boundary that adoption is only for operator-owned project roots. Preferably verify the target looks like a project repo/worktree before writing `.vcf/`.
