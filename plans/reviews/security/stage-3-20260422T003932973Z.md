# security — Stage 3 — NEEDS_WORK

## Summary

Security boundary enforcement is mostly intact in the reviewed diff, but two meaningful gaps remain: the new adoption path writes and registers arbitrary operator-supplied directories without any visible containment against symlink-targeted writes or duplicate/root-path collision checks, and the new project-scope release-state transition updates privileged lifecycle state with no visible authorization check in the changed execution path itself. Config default endpoint enforcement for review execution is directionally sound and includes trust-level gating, but the added bypass-style adoption feature weakens provenance and boundary assumptions enough to require remediation before this stage can pass cleanly.

## Findings

- **warning** — `src/cli.ts:787` — The new `vcf adopt` path accepts an arbitrary filesystem path, creates `absRoot/.vcf`, and opens `absRoot/.vcf/project.db` after only checking that `absRoot` currently exists and is a directory. There is no visible defense here against a pre-existing `.vcf` path being a symlink or against the target DB path resolving outside the adopted project root. This weakens the filesystem trust boundary for a privileged local action that writes tracking state and could clobber or create files outside the intended project boundary if an attacker can influence the adopted directory contents.
  - required: Before creating or opening `.vcf/project.db`, canonicalize and verify the path chain (`realpath`/lstat checks), reject symlinked `.vcf` directories or DB files, and ensure the final DB path remains inside the canonical project root. Add regression tests for symlinked `.vcf` and symlinked `project.db` cases.
- **warning** — `src/cli.ts:829` — `runAdopt` registers the adopted project into the global registry via `upsertProject(globalDb, { name: projectSlug, root_path: absRoot, state: existing?.state ?? state })`, but the reviewed diff shows no collision or ownership check on existing global rows for the same root path or slug before this privileged registry update. Because adoption is explicitly a bypass mode for pre-existing projects, weak registry-boundary enforcement risks phantom project identity overlap, stale state aliasing, or unintended reassignment of lifecycle metadata across trust zones the operator may assume are distinct.
  - required: Enforce explicit duplicate/root-path collision handling in the adopt flow: reject conflicting existing registrations unless a verified re-adoption of the same canonical root is intended, and surface a clear confirmation path when overwriting registry metadata. Add tests for slug collision and canonical-root collision behavior.
- **warning** — `src/tools/ship_release.ts:240` — The diff adds a direct privileged state transition (`project.state = 'shipped'` plus `setProjectState(..., 'shipped')`) after release success, but no authorization or scope enforcement is visible in the changed path itself. Stage 3 requires privileged actions to be visibly gated on live paths, not assumed from surrounding architecture. If `ship_release` registration lacks a strong confirm+authz boundary elsewhere, this change would silently extend that weakness into project lifecycle state mutation and global registry mutation.
  - required: Make the privilege boundary explicit in code near this state transition or in the tool registration path: document and enforce the authn/authz/confirmation requirements for invoking `ship_release`, and add an integration test demonstrating unauthorized callers cannot trigger the shipped-state transition even when confirmation semantics are attempted.
