# security — Stage 1 — BLOCK

## Summary

Security scope is materially clearer than before: the diff names new high-value assets (project lessons, global lessons mirror, response log, lifecycle report JSON/markdown), new entry points (`project_init_existing`, `lesson_log_add`, `lesson_search`, `lifecycle_report`), and the model-provider boundary for review execution and narrative reporting. However, Stage 1 cannot pass because one major trust boundary is still wrong at the design level: any project-scoped session can query the shared global lessons DB across projects, but the docs and code do not define that as a privileged cross-project read surface. In addition, narrative `lifecycle_report` sends audit/review/response/lesson metadata to a configured endpoint, but the surfaced obligation/routing guidance is much weaker than the equivalent warning added for `review_execute`, leaving provider-handling scope under-specified for a new outbound data path.

## Findings

- **blocker** — `src/tools/lesson_search.ts:24` — Cross-project lesson data is exposed as a normal project-scope read surface without an explicit trust boundary. `lesson_search` accepts `scope: "global" | "all"` and, when selected, opens the shared global lessons DB and returns rows from all mirrored projects (`src/tools/lesson_search.ts:24`, `src/tools/lesson_search.ts:71`, `src/tools/lesson_search.ts:77`). README confirms the mirror is cross-project and queryable from any project session (`README.md:181`). Lessons are an identified sensitive asset because they may contain redacted-but-still-sensitive operational and design observations. Stage 1 fails here because actor/privilege classes are collapsed: a project-scoped session for repo A is implicitly trusted to read lessons from repos B/C with no modeled authorization boundary or opt-in read control.
  - required: Make the cross-project boundary explicit and enforced. Either restrict project-scope `lesson_search` to the current project by default with a separate privileged/global surface for cross-project reads, or document and gate global reads as an operator-level capability with verifiable controls. Carry forward the global lessons DB as a distinct high-value asset and define which actors may read project-scoped vs universal lessons.
- **warning** — `src/tools/lifecycle_report.ts:623` — The new narrative lifecycle-report path externalizes a broad project snapshot to a model endpoint, but the obligation/data-routing scope is under-described compared with the review path. The prompt builder serializes whole lifecycle sections, including audit activity, review history, response-log metadata, builds, and lesson titles/tags (`src/tools/lifecycle_report.ts:623`, `src/tools/lifecycle_report.ts:652`, `src/tools/lifecycle_report.ts:672`). README documents a strong routing warning for `review_execute` (`README.md:223`) but the lifecycle-report section only mentions redaction and model defaults, not the sensitivity class of data sent to providers (`README.md:172`). Stage 1 needs that outbound provider boundary stated clearly enough that later security/production stages know what obligations apply.
  - required: Add explicit scope documentation for `lifecycle_report` narrative mode matching the review routing note: name the data classes sent off-host, state that redaction is not equivalent to confidentiality, and tie endpoint choice to trust/residency obligations. Carry this boundary forward for later provider-data-handling review.
