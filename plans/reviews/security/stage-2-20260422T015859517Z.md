# security — Stage 2 — NEEDS_WORK

## Summary

Stage 2 threat review found one realistic high-impact AI abuse path partially but not structurally mitigated: review payloads containing sensitive diffs/carry-forward data can still be sent to any configured non-public provider selected via config defaults or explicit endpoint, with trust gating only blocking `public` endpoints. The diff improves operator safety around `vcf adopt`, adds endpoint/default validation, and redacts `ship_release` tails before persistence, which bounds several abuse paths. But the model/provider trust boundary remains permissive enough that prompt/context leakage and provider-side retention exposure are still credible for high-value review content, especially now that endpoint choice can be made implicit via `config.defaults.review`. No direct tool-to-host privilege-escalation path is introduced in this diff, but AI-specific exfiltration risk remains partially mitigated rather than closed.

## Findings

- **warning** — `src/tools/review_execute.ts:106` — Model/provider trust boundary is only enforced at `trust_level === "public"`; any configured `local` or `private` endpoint may receive full redacted review payloads, including scoped diffs and carry-forward notes, with no per-run data-sensitivity gate or operator confirmation. The new fallback to `config.defaults.review.endpoint` makes this route easier to trigger implicitly when callers omit `endpoint`, increasing the chance that sensitive review context is shipped to a provider the operator did not intend for that run. Abuse path: attacker or careless operator causes review of credential-heavy/regulatory codebase -> `review_execute` resolves default endpoint -> full prompt bundle is transmitted to external provider classified as non-public -> provider retention/logging/model access exposes confidential repo context. Evidence: endpoint default resolution and provider selection in `src/tools/review_execute.ts:100-116`, model call in `src/tools/review_execute.ts:151-174`, config defaults schema in `src/config/schema.ts:207-245` and `src/config/schema.ts:310-320`, README encouragement to set implicit defaults in `README.md:197-244`.
  - required: Add a structural control for sensitive review payload routing: e.g. require an explicit per-call opt-in/confirm token for non-local endpoints, or add a stronger trust tier/policy that can restrict review payloads to local endpoints only unless the run is explicitly marked exportable. At minimum, document and enforce a machine-checkable distinction between local-hosted and externally retained providers rather than treating all non-public providers as equally safe.
