# security — Stage 2 — NEEDS_WORK

## Summary

Threat model is mostly clear and the high-risk chains are concrete: (1) cross-project lesson disclosure remains an intentional single-operator workstation boundary, bounded by `scope` and the new `global_db_path: null` disable path; (2) prompt-injection against `lifecycle_report` narrative mode is partially mitigated because untrusted project text is serialized into LLM prompts but model output does not trigger tools or host actions; (3) tool misuse / agent / infrastructure privilege escalation are currently bounded because no new AI surface auto-executes model output; (4) the main remaining abuse path is data exfiltration through model-provider boundaries, especially now that `review_execute` and `lifecycle_report` can resolve endpoints implicitly from config defaults. Highest-value targets are review diffs, response-log history, audit/build/review metadata, and lesson data. Highest-risk realistic chain: config/default routing change or careless endpoint selection -> full review/lifecycle prompt bundle sent to a non-local/provider-backed endpoint -> off-host retention/exposure. That provider-boundary control is documented but not fully enforced in code.

## Findings

- **warning** — `src/tools/review_execute.ts:101` — AI/provider trust boundary is only partially enforced for review runs. The new default-resolution path allows `review_execute` to omit `endpoint` and fall back to `config.defaults.review.endpoint`, while the runtime gate only blocks `trust_level === "public"`. Per the updated docs, non-public/proxy endpoints can still forward the full review bundle off-host; the prompt here includes the scoped diff, prior responses, carry-forward, and review history. Abuse path: operator or config drift changes `defaults.review.endpoint` to a remote/proxy target, a later run omits explicit routing, and sensitive code/review context is exfiltrated without per-run confirmation.
  - required: Require an explicit opt-in for any non-local review endpoint selected via defaults, not just `public` ones, or force explicit `endpoint` on `review_execute` whenever the resolved endpoint is not truly local. At minimum, add a hard gate tied to `endpoint.trust_level != "local"` plus an auditable override.
- **warning** — `src/tools/lifecycle_report.ts:621` — Narrative lifecycle reporting has the same residual exfiltration path. `runNarrativeCore` resolves `config.defaults.lifecycle_report.endpoint` and only requires override consent for `trust_level === "public"`, even though the tool serializes broad project-derived data (audit, reviews, responses, builds, lessons metadata) into outbound prompts. Abuse path: malicious or careless endpoint configuration on a provider-backed but non-`public` route causes project history to leave the workstation during a normal `lifecycle_report` run.
  - required: Apply the same non-local endpoint gating to narrative mode that sensitive review traffic needs: require explicit approval for any endpoint that is not local, or restrict defaults-driven narrative routing to `trust_level: local` unless the caller passes an override flag.
