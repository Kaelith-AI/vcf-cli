# security — Stage 2 — NEEDS_WORK

## Summary

The update introduces a new 'adopt' feature and a configuration-based routing system for LLM endpoints. While the core logic is structurally sound, the 'adopt' feature introduces a significant trust boundary risk: it allows the CLI to write to arbitrary directories and modify a global registry based on user-provided paths. Additionally, the routing logic for LLM endpoints, while convenient, creates a potential for 'silent' model degradation or redirection if the config file is compromised, though this is a lower risk than the filesystem operations.

## Findings

- **warning** — `src/cli.ts:787` — The `runAdopt` function accepts a `path` argument and performs filesystem operations (mkdir, writeFile) within that path. While this is a CLI tool running in the user's context, the lack of validation against the current working directory or a restricted set of paths could allow a user (or a script calling the CLI) to create `.vcf` directories and database files in sensitive system locations if the CLI is run with elevated privileges.
  - required: Add a check to ensure the `path` is within the user's home directory or a designated workspace directory, or explicitly document that this tool is intended to be run with the same privileges as the user and should not be run as root/sudo.
- **warning** — `src/cli.ts:818` — The `runAdopt` function updates a global registry database located at `~/.vcf/vcf.db`. If an attacker can influence the `name` or `path` passed to `runAdopt`, they could potentially pollute the same global registry used by other projects, leading to confusion or redirection of VCF tools to malicious project roots.
  - required: Implement a validation check to ensure the `name` (slugified) is unique or that the user has ownership of the path being adopted.
- **info** — `src/config/schema.ts:230` — The introduction of `defaults` in `ConfigSchema` allows the system to route requests to different endpoints based on the tool being used. If `config.yaml` is compromised, an attacker could redirect specific high-value tools (like `review_execute`) to a malicious endpoint that captures the prompt and context, and then returns a fake 'PASS' verdict. This is a 'silent' redirection risk.
  - required: No structural change required, but this should be noted as a residual risk in the threat model for the operator.
