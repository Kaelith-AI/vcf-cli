# security — Stage 4 — BLOCK

## Summary

The implementation of the `vcf adopt` command introduces a critical Path Traversal vulnerability. The command allows an operator to specify an arbitrary path, which is then used to create a `.vcf` directory and a `project.db` SQLite file. While there is basic validation that the path exists and is a directory, there is no validation to ensure the path is not a sensitive system directory (e.g., `/etc`, `/var/log`, or other users' home directories). An attacker or a compromised operator could use this to write a `.vcf/project.db` file into arbitrary locations on the filesystem, potentially overwriting existing files or creating markers in sensitive directories. Furthermore, the `runAdopt` function uses `absRoot.split('/').pop()` to derive a project name, which is fragile and potentially misleading. The `templatesDir` fix, while improving reliability, uses a hardcoded `PACKAGE_ROOT` (redacted in diff) which may not be portable across all environments, though this is is a lower risk.

## Findings

- **blocker** — `src/cli.ts:798` — Path Traversal / Arbitrary File Write: The `runAdopt` function accepts a `path` argument from the CLI, resolves it to an absolute path, and then proceeds to create a directory `.vcf` and a file `project.db` within that path without any restriction on where that path can be located. This allows writing to any directory the process has permissions to write to.
  - required: Implement a strict allowlist of directories or ensure the path is constrained to a specific workspace root. At minimum, the tool should prevent adoption of system directories (e.g., /etc, /var, /root) and check for existing files that would be conflict with the .vcf directory creation.
- **warning** — `src/cli.ts:818` — Fragile Project Name Derivation: The logic `absRoot.split('/').pop()` to derive the project name is platform-dependent (fails on Windows `\`) andes assumes the directory name is the project name. This can lead to inconsistent project naming in the global registry.
  - required: Use `path.basename(absRoot)` for cross-platform compatibility.
- **info** — `src/util/templates.ts:41` — Potential Portability Issue: The `templatesDir` function now relies on a `PACKAGE_ROOT` constant. If this constant is resolved incorrectly at runtime in certain bundled environments, the tool will fail to find its templates.
  - required: Ensure `PACKAGE_ROOT` is dynamically resolved using the `findPackageRoot` helper provided in the same file, rather than being a static constant.
