# security — Stage 4 — NEEDS_WORK

## Summary

Reviewed secure-coding reality for the scoped changes around project adoption, review endpoint/model defaulting, release-state transition, and template path resolution. Input validation is generally present on newly added CLI/config paths, and the review endpoint trust gate still executes after default resolution. However, one sensitive implementation gap remains: `ship_release` now mutates authoritative project state to `shipped` before confirming the registry mirror succeeded, then silently suppresses registry-update failures. That creates inconsistent security/operational state across trust boundaries after a release and can mislead downstream tooling about project lifecycle status. No direct injection, crypto, or auth bypass issue was evident in the diff; main risk is unsafe state-handling in a security-relevant transition.

## Findings

- **warning** — `src/tools/ship_release.ts:248` — Successful release now updates the authoritative per-project DB to `state='shipped'` and then attempts to mirror that state into the global registry, but any registry failure is swallowed (`catch { /* non-fatal */ }`). This creates a real split-brain state transition across boundaries: project-local state says shipped while global registry cache may still say something else. For tooling that relies on the registry for portfolio/status decisions, this is an unsafe partial commit on a security-relevant lifecycle boundary, and the code provides no audit signal or repair path when the mirror fails.
  - required: Do not silently suppress registry mirror failure. At minimum, emit a security-relevant warning/audit record when `setProjectState` fails so operators can detect and repair divergence. Prefer making the two writes consistent by explicitly reconciling/repairing the registry cache after failure or by structuring the transition so downstream readers cannot treat stale registry state as authoritative.
